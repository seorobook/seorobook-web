# 서재 기반 SNS — 제품 메모

> 이 문서는 기획·기능·질문을 계속 업데이트하며 진행합니다.

---

## 1. 문제 인식

- 정보 홍수, 너무 많은 사람들의 목소리, 둠 스크롤링으로 시간·에너지 낭비
- **목표**: 안전한 나만의 SNS 공간

---

## 2. 제품 컨셉

- **시작점**: 독서 기록 앱
- **공간**: 로그인 시 “나만의 서재” (테이블명 **libraries**, 1유저 1서재 unique)
- **톤**: 옛날 싸이월드 느낌 — 나만의 공간 + 한정된 주변 사람/이웃과의 나눔

## 3. 핵심 기능 (구상)

### 3.0 기본 화면(/app) 합의 (구현 방향)

- 로그인 후 기본 진입은 **`/app`** (레거시 “Your Libraries” 선택 화면은 제거).
- `/app`은 더 이상 “맵 편집기”가 아니라, **서재 공간(가운데)** + **기능 패널(오른쪽/모바일 중심)**을 가진 **기본 홈**으로 사용.
- 서재 공간 렌더링은 **PlayApp 기반**을 우선 채택.
  - 기본은 **싱글/로컬(가볍게)**.
  - “방문” 세션에 들어갈 때만 **멀티/보이스 ON**.
- UI 전환은 상태 전환보다 **URL 라우팅 중심**으로 설계(모바일/딥링크/캐시 고려). Next.js의 parallel routes(`@`), route groups(`()`), optional catch-all(`[[...]]`) 등을 활용.

### 3.1 서재 & 독서 기록

- 서재에 **책장 오브젝트** 배치
- 책장 클릭 → **읽은 책 기록**, **감상/리뷰** 작성
- 리뷰 중 **일부만 선택적으로 공개** 가능

### 3.2 관계 모델: "서로"

- **이웃 이름 = "서로"** (프로젝트명 서로북과 맞춤). **서로 신청** → 상대가 수락하면 관계 성립. (DB: 관계 1쌍당 **1행** — 인증·보안 일관성.)
- **서로 간 메신저(DM)**: 수락된 서로끼리는 **다이렉트 메시지** 주고받기 가능.
- **폐쇄형**: 상호 수락된 서로에게만 선택 공개 리뷰가 피드 등으로 유통.
- **방문** (아래 3.5와 동일 개념):
  - **방문 전에** 서로 신청·수락을 끝내거나,
  - **방문 희망**을 먼저 할 수도 있음: 서재 주인에게 명단이 가고, 주인이 수락·스케줄해서 방문 일시 확정.
- **불특정 다수**: **컨텐츠**를 **피드에 띄우기**(피드 발행). 컨텐츠는 (나중에 그래프 DB 마이그레이션 예정) 떠돌다가 **추천/매칭**으로 사람들 **피드**에 도달. **서로** 우선 노출, 비서로 중 "피드 수신 허용" 유저는 광범위 노출. **받은 것**으로 보지 않음.
- **컨텐츠(리뷰 등)를 열어본 사람** → 관심 생기면 **놀러가기 신청** 가능. 언제·왜 놀러가고 싶은지 어필.
- **피드 발행** 하루 횟수 = **최소 1회 + 서로 수에 비례**. 캐시로 추가 발행 구매 가능.

### 3.3 공개 리뷰의 유통

- 내가 공개한 리뷰 → 추천/선별 후 **이웃의 피드**로 전달 (마구잡이 피드 아님).

### 3.4 피드 (메일함·뉴스 통일)

- 서재에 **메일함 오브젝트** 클릭 → **피드** 화면.
- **피드**: 전통 SNS보다 **큐레이션된** 형태. **배치 단위**로 묶임.
  - 예: **컨텐츠** 10개를 묶어 한 배치로 주면 → 그 시점 **피드 1개**.
  - **한 피드(배치) = 컨텐츠 5~10개**. 컨텐츠 = 글 한 편 (길 수 있음, 책 리뷰 등).
  - **피드 개수** 사용자 설정: **최소 1, 최대 100** (한 번 열람 시 배치 수).
- **사용자 기준 24시간** 1회 무료 오픈. 추가 열람 **유료(캐시)**.
- **매칭·추천** 선별, **AI 요약** → 관심 생기면 **원문** 열기.
- **피드 발행 컨텐츠**도 피드에 도달. **수신 거부** 가능.
- 선별 기준: 서로 글, 관심 있을 만한 것. **초반에는 책 리뷰 위주**.

### 3.5 방문 (독서 모임 통일)

- **방문 = 독서 모임**. 개념 통일: **"방문"**.
- **방문**은 **여러 명**일 수 있음. 방문 희망 → 서재 주인 수락·스케줄 → **방문 OK**된 사람은 **허락된 시간 동안** 서재에 입장, **음성** 대화. **서재 주인이 음성 mute** 가능.
- **약속 시간**: 주인과 초대 대상(들) **합의**로 결정.
- **기본 1명** 초대, 캐시로 **인원 확장**. 참여자는 무료.
- 필요 시 게시판 오브젝트로 일정/초대 관리.
- 구현 합의:
  - 방문은 **방문 세션(약속/초대/시간 제한)** 기반으로 반드시 분리.
  - 보이스는 **방문 세션에서만** 사용.
  - 방문자는 초기 버전에서는 **입장 + 보이스로 대화만** (오브젝트 상호작용/추가 기능은 이후 확장).

### 3.6 리뷰·독서 기록 정책

- **리뷰 기본 비공개**. 폐쇄형 SNS이므로 초반에는 **독서 기록에 더 힘** 줌.

### 3.7 UX 원칙

- **걸어다니는 느낌 유지**. 클론의 서재(스페이스) 이동·오브젝트 배치 감각 유지.
- 클론에 있는 기능 중심으로 **과하게 확장하지 말고 최소한만** 구현 (일단).

### 3.8 BM (비즈니스 모델)

- **캐시 아이템 구매**
- **집사(Butler)** 구매 시: AI가 책 리뷰 보조, 질문·정리·검색 등.
- **피드 추가 열람** 유료, **방문 호스팅 인원 확장** 유료, **피드 추가 발행** 유료.

## 7. DB·오브젝트 설계 초안

MVP 기능 목록을 위한 테이블·엔티티 초안. **용어 통일**: 서재 = **libraries**, 방문 = 독서 모임, 뉴스 = **피드**.

### 7.1 유저·서재

| 테이블 | 용도 | 비고 |
|--------|------|------|
| **users** | Neon Auth 관리. 별도 테이블 없이 auth user id 사용. | — |
| **profiles** | 닉네임, 아바타, skin 등. | 클론에 있음. 확장. |
| **libraries** | 서재 1개 = 1 row. **owner_id = 유저, UNIQUE** (1유저 1서재). map_data에 서재 배치·오브젝트 위치. | 기존 realms 마이그레이션 시 **libraries**로 변경 권장. |

### 7.2 오브젝트(책장·메일함·물병)

- **맵 위 오브젝트**: `libraries.map_data`(JSON) 안에 **타일/오브젝트 타입**으로 배치 (예: `object: "bookshelf"`, `"mailbox"`, `"bottle"`). 별도 오브젝트 테이블 없이, 클릭 시 **해당 서재 주인(owner_id) 기준**으로 백엔드 데이터 조회.
- 책장 클릭 → 해당 유저의 독서 기록·리뷰.
- 메일함 클릭 → 해당 유저의 **피드**.
- 물병은 “띄우기” 액션만 맵/오브젝트에 두고, 발행·추천/매칭은 아래 테이블.

### 7.3 독서 기록·리뷰

| 테이블 | 용도 | 비고 |
|--------|------|------|
| **books** (또는 reading_records) | id, user_id, title, author, read_at, created_at | 읽은 책 한 줄. |
| **reviews** | id, user_id, book_id, content, visibility (private \| public \| 서로만 등), created_at | 감상. 기본 비공개. |

### 7.4 서로(관계)

| 테이블 | 용도 | 비고 |
|--------|------|------|
| **relationships** (또는 sero) | id, requester_id, addressee_id, status (pending \| accepted), created_at | **1쌍당 1행** (인증·보안). (user_a_id, user_b_id) 정렬 규칙으로 unique. 수락 시 accepted. |
| **direct_messages** | id, sender_id, receiver_id, content, created_at | **서로 간 DM**. 수락된 서로끼리만. |

### 7.5 방문 (독서 모임 통일)

| 테이블 | 용도 | 비고 |
|--------|------|------|
| **visits** | id, library_id, host_id, scheduled_at, max_participants, status | **방문** 1건. 여러 명 가능. 기본 1명, 캐시 확장. |
| **visit_invitees** | visit_id, user_id, status, message(이유·희망 시각) | 방문 희망·수락 시 허락된 시간에 입장·음성. 주인 mute 가능. |

### 7.6 피드 (메일함·뉴스 통일)

| 테이블 | 용도 | 비고 |
|--------|------|------|
| **feed_batches** | id, user_id(수신자), opened_at, created_at | 피드 1개 = 배치 1개. 한 배치 = 컨텐츠 5~10개. |
| **feed_batch_items** | id, feed_batch_id, source_type (review \| bottle), source_id, summary, sort_order | AI 요약. 원문 source_type+source_id. |
| **user_feed_settings** | user_id, min_feed_count, max_feed_count | 피드 개수 사용자 설정 **1~100**. |
| **feed_opened_at** | user_id, last_opened_at | 24h 1회 등 정책용. |

### 7.7 피드 발행 (컨텐츠 띄우기)

| 테이블 | 용도 | 비고 |
|--------|------|------|
| **bottle_messages** (또는 feed_publications) | id, sender_id, content, sent_at | **컨텐츠** 띄우기만. 받은 것 테이블 없음. 나중에 그래프 DB 마이그레이션. |
| **bottle_opt_out** | user_id | 피드 수신 거부. 수신 허용 비서로는 추천으로 광범위 노출. |
| 발행 한도·캐시 | 최소 1회 + 서로 수. 당일 집계. 캐시 추가 = transactions. | 컨텐츠 열어본 뒤 관심 시 방문 신청(visit_invitees) 가능. |

### 7.8 BM·캐시

| 테이블 | 용도 | 비고 |
|--------|------|------|
| **user_balances** (또는 cash) | user_id, balance | 캐시 잔액. |
| **transactions** | id, user_id, amount, product_type (feed_extra \| visit_capacity \| feed_publish_extra \| butler 등), reference_id, created_at | 충전·소비 내역. |

### 7.9 정리

- **신규 테이블 후보**: books, reviews, relationships, direct_messages, visits, visit_invitees, feed_batches, feed_batch_items, user_feed_settings, feed_opened_at, bottle_messages, bottle_opt_out, user_balances, transactions.
- **테이블명**: 서재 = **libraries** (owner_id UNIQUE). 기존 realms → 마이그레이션 시 libraries 권장.
- **인덱스·제약**: user_id, library_id, (sender_id, sent_at), (user_id, last_opened_at) 등 조회 패턴 보고 추가.

### 7.10 용어 정리

- **컨텐츠**: 글 한 편 (리뷰, 피드에 띄운 글 등). "편지" 대신 **컨텐츠** 사용.
- **피드**: 배치 단위의 큐레이션된 목록. 한 피드 = 컨텐츠 5~10개.
- **피드 발행**: 컨텐츠를 피드에 띄우는 기능. (기존 "물병" 표현 제거, DB 테이블명 bottle_* 는 유지 가능.)
